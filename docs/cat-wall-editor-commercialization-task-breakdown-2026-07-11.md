# 猫墙 3D 编辑器商业化任务拆解（2026-07-11）

> **2026-07-16 状态说明：** 本文仍可作为商业化 MVP 看板来源；最新实现状态和已完成基础能力请先看 `docs/cat-wall-editor-handoff-2026-07-16.md`。其中施工图基础功能已实现，P0 后续应聚焦 SKU、报价、项目/客户信息、施工包增强和规则提示产品化。

> 来源：`docs/cat-wall-editor-commercialization-plan-2026-07-09.md`
>
> 拆分原则：开发任务与数据录入/业务内容任务分开管理；首期只围绕“真实 SKU -> 设计方案 -> 自动报价 -> 施工图 PDF -> 客户确认 -> 安装执行”的最小闭环。

## 1. 范围和优先级

### P0：内部成交工具 MVP（第 1-4 周）

必须完成：

- 真实 SKU Catalog。
- 项目与客户基础信息。
- 商品计价、人工调整和报价单。
- 施工图 PDF 与 BOM。
- 方案列表、复制、重命名。
- 安装规则提示优化。
- 10 套模板和 10 个真实场景验收。

### P1：客户预览与确认（第 5-6 周，P0 验收后启动）

- 客户只读预览页。
- 分享链接。
- 客户确认或留言。
- 转化事件记录。

### 本轮不做

- 在线支付、完整订单系统。
- 多人实时协作、复杂权限、历史版本。
- 多城市自动计价、完整库存系统。
- AI 自动布局和大型数据看板。
- SaaS 多租户和供应链平台。

#### P0 开发验收标准

1. 旧版项目 JSON 和 IndexedDB 草稿能打开，已有场景、组件和禁区不丢失。
2. 30-50 个真实 SKU 可一次导入；无效数据必须指出具体行、字段和原因，不允许静默忽略。
3. 同一方案的报价面板、报价 PDF 和 BOM 中商品数量及金额一致。
4. 商品小计、安装费、运费、优惠的计算结果与人工核算误差不超过 5%；金额计算不得出现浮点小数误差。
5. 项目复制后生成新 ID，场景内容完整，原项目不被后续修改影响。
6. 每面墙的施工图包含组件编号、位置尺寸和打孔信息；安装人员只看施工包即可指出安装顺序和所需配件。
7. 每条阻断级规则提示均包含“对象/位置、失败原因、可执行建议”。
8. 10 个真实场景全部能从建项目走到报价和施工包；平均初版方案时间小于 15 分钟。
9. `npm run test` 和 `npm run build` 通过。

#### P1 开发验收标准

1. 未持有分享令牌的用户无法读取项目，分享链接可撤销。
2. 客户页不暴露客户联系方式、内部成本、毛利、设计师编辑入口或原始 JSON。
3. 客户可在手机端完成查看、确认或提交修改意见。
4. 重复点击确认不会生成重复记录，确认记录能定位到确定的项目和报价快照。
5. 分享打开、确认、请求修改三个事件可按项目查询或导出。

## 4. 数据录入和业务内容任务

这些任务由商品/运营主责，产品负责人和安装顾问评审。开发只负责数据结构、导入工具和错误反馈，不代替业务人员决定价格、承重和施工规则。

### P0：必须与开发并行

| ID | 任务 | 录入/整理内容 | 前置依赖 | 估算 | 建议负责人 |
|---|---|---|---|---:|---|
| DATA-01 | 建立数据字典和唯一编码规则 | SKU、模板、配件包、规则、案例的 ID 格式；单位；必填项；枚举值；字段负责人 | 无 | 1 天 | 产品 + 商品运营 |
| DATA-02 | 整理 30-50 个核心 SKU | SKU、名称、分类、尺寸、售价、上下架状态、模型映射；补齐颜色、材质、承重、适用墙体、是否打孔 | DATA-01 | 3-5 天 | 商品运营 |
| DATA-03 | 整理 SKU 视觉和 3D 资产 | 主图、缩略图、GLB、统一朝向/比例/原点；低频 SKU 指定占位模型；记录缺失和待制作项 | DATA-01、DATA-02 | 3-10 天，可并行 | 3D 资产 + 商品运营 |
| DATA-04 | 定义首版报价规则 | 币种；含税口径；商品价；安装费、运费、优惠的输入范围；有效期；待确认项；四舍五入规则 | DATA-01 | 1-2 天 | 业务负责人 + 财务/销售 |
| DATA-05 | 建立配件包和 BOM 映射 | 每个 SKU 对应的支架、螺丝、膨胀件、连接件、耗材、数量和替代件 | DATA-01、DATA-02 | 2-4 天 | 商品运营 + 安装顾问 |
| DATA-06 | 建立安装规则表 | 墙体适配、承重下限、跳跃距离、起点高度、安全缓冲、打孔禁区；错误文案和修复建议 | DATA-01、DATA-02 | 2-4 天 | 安装顾问 + 产品 |
| DATA-07 | 定稿报价单和施工包内容 | 报价单字段、公司信息、条款、有效期文案；施工图页序、图例、注意事项、复核清单和签字区 | DATA-04、DATA-05 | 1-2 天 | 产品 + 设计 + 安装顾问 |
| DATA-08 | 制作 10 套高质量模板 | 覆盖单墙/双墙、多猫、幼猫/老猫、低/中/高预算、少打孔等；记录适用尺寸和替换规则 | DATA-02、DATA-03、DATA-06 | 3-5 天 | 设计师 + 商品运营 |
| DATA-09 | 整理 10 个真实验收案例 | 客户信息脱敏；墙面尺寸、障碍物、墙体、猫属性、预算；期望组件、价格范围和关键施工风险 | DATA-02、DATA-04、DATA-06 | 2-3 天 | 产品 + 设计师 |
| DATA-10 | 执行内测并录入结果 | 每案例开始/结束时间、人工核价、错误、返工点、安装人员反馈、最终是否通过 | DEV-12、DATA-09 | 2-3 天 | QA + 设计师 + 安装顾问 |

#### P0 数据验收标准

1. SKU 表中 100% 条目有唯一 SKU、名称、分类、长宽高、销售价、上下架状态和模型/占位模型。
2. 尺寸统一用毫米录入，重量统一用千克，金额统一用人民币元；不得在同一列混用单位。
3. 每个上架 SKU 均有资产状态、配件包状态和安装规则状态；未完成项必须显式标记，不得留空冒充可用。
4. 报价规则包含至少 5 个算例：无附加费、安装费、运费、优惠、待确认项；算例有人工确认结果。
5. 每个打孔类 SKU 均有配件包；每个配件项均有名称、规格和数量。
6. 每条安装规则均包含规则 ID、适用范围、阈值、级别、原因文案和建议文案，并经安装顾问签字/留痕确认。
7. 10 套模板均能在当前编辑器中打开，无越界或阻断级冲突，且有明确预算档和适用场景。
8. 10 个案例均有预期报价范围和至少一个安装校验点，可作为通过/失败的判断依据。

## 5. 依赖关系与并行方式

```text
DATA-01 数据字典
  +-> DATA-02 SKU -------------------> DEV-02 Catalog
  |      +-> DATA-03 资产
  |      +-> DATA-05 配件 -----------> DEV-08 施工包 / DEV-09 BOM
  |      +-> DATA-06 安装规则 --------> DEV-10 规则引擎
  |      +-> DATA-08 模板 ------------> DEV-11 模板复用
  |
  +-> DATA-04 报价规则 -> DEV-01 数据模型 -> DEV-03 计价 -> DEV-04 面板 -> DEV-05 报价单
  |
  +-> DATA-09 验收案例 ---------------------------------------------> DEV-12 / DATA-10

DEV-06 项目信息 -> DEV-07 项目管理

P0 全部验收 -> DEV-13 分享 API -> DEV-14 客户预览 -> DEV-15 确认/留言
```

并行建议：

- 第 1 周：DATA-01 至 DATA-07 并行推进；开发完成 DEV-01，并提前搭建 DEV-02、DEV-06。
- 第 2 周：开发并行做 Catalog/报价链路和项目管理；数据侧完成资产、配件、规则和前 5 套模板。
- 第 3 周：开发并行做施工包、BOM、规则引擎和模板复用；数据侧完成全部模板和案例。
- 第 4 周：只做 10 个案例内测、修复阻断问题和指标复核，不再新增范围。
- 第 5-6 周：只有 P0 达标后才进入客户预览、分享和确认；否则继续修复成交与施工闭环。

工作量汇总：

- P0 开发：约 25-29 人日；1 名开发需要约 5-6 周，2 名开发在依赖顺畅时可压到约 3-4 周。
- P1 开发：约 10-12 人日；建议至少 1 名前端和 1 名全栈/后端并行。
- P0 数据与业务内容：约 20-39 人日，其中 3D 资产差异最大；商品运营、设计师和安装顾问必须并行，不能串行等待。
- P1 数据与业务内容：约 3-4 人日。

## 6. 首期完成门槛

P0 只有同时满足以下条件才算完成：

1. 数据侧：30-50 个 SKU、10 套模板、10 个验收案例全部通过数据验收。
2. 开发侧：DEV-01 至 DEV-12 全部通过各自验收，自动化测试和构建通过。
3. 业务侧：10 个案例完整走通，平均初版方案小于 15 分钟，报价误差不超过 5%。
4. 交付侧：安装顾问可仅凭施工包完成信息复核，未发现阻断施工的缺失项。
5. 范围侧：没有把支付、复杂权限、AI 自动布局等 P2 项目夹带进 P0。

## 7. 建议的任务看板字段

每张卡至少维护：

- `任务 ID`：如 `DEV-04`、`DATA-06`。
- `类型`：开发 / 数据录入 / 业务评审 / 测试。
- `优先级`：P0 / P1 / Backlog。
- `负责人` 和 `评审人`。
- `前置依赖`。
- `交付物链接`：代码、数据表、设计稿或测试记录。
- `验收标准`：从本文复制对应的可判定条目。
- `状态`：待开始 / 进行中 / 待评审 / 被阻塞 / 已完成。
- `阻塞原因`：缺数据、缺业务决策、缺资产、技术问题，四类分开记录。
