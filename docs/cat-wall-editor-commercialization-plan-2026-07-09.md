# 猫墙 3D 编辑器商业化计划书（2026-07-09）

> **2026-07-16 状态说明：** 本文是商业化方向和路线图。当前实现进度、已完成能力和最新技术交接请先看 `docs/cat-wall-editor-handoff-2026-07-16.md`；任务拆解请看 `docs/cat-wall-editor-commercialization-task-breakdown-2026-07-11.md`。

## 1. 项目概述

猫墙 3D 编辑器当前已经具备从空间搭建、组件摆放、禁区避让、距离测量、施工图生成到 BOM 清单的核心能力，已经不再只是展示型 demo，而是一个接近“可成交、可交付”的垂直设计工具。

商业化阶段的目标，不是继续单纯强化 3D 编辑能力，而是把编辑器升级为一套面向真实业务的“猫墙方案配置与交付系统”：

- 面向客户：快速生成可理解、可确认、可下单的猫墙方案。
- 面向设计师/客服：快速出方案、改方案、报价、解释限制。
- 面向安装师傅：生成可执行的施工图、打孔坐标、组件清单和安装顺序。
- 面向经营者：管理商品、模板、价格、库存、订单和转化数据。

一句话定位：

> 一个连接“猫墙设计、报价成交、施工交付”的垂直 3D 配置器。

## 2. 商业化目标

### 2.1 短期目标：让工具能支撑成交

在 4-6 周内，把当前编辑器打磨成可以服务真实客户咨询的内部工具。目标是让客服或设计师能够用它完成：

- 录入客户墙面信息。
- 根据猫数量、预算、空间限制生成方案。
- 快速拖拽调整组件。
- 自动校验禁区、间距、组件冲突。
- 生成报价单和施工图。
- 发给客户确认。

短期成功标志：

- 能用编辑器完成 10 个真实客户方案。
- 每个方案从“客户给尺寸”到“初版报价”控制在 15 分钟以内。
- 施工图能被安装师傅看懂并执行。
- 报价总价与人工核算误差控制在 5% 以内。

### 2.2 中期目标：让客户能半自助设计

在 2-3 个月内，推出面向客户的简化模式，让客户可以选择模板、替换组件、查看价格，并提交咨询/下单意向。

中期成功标志：

- 客户能在无指导情况下完成一个基础方案。
- 设计提交转化为咨询的比例达到 15%-25%。
- 客服基于客户草稿二次修改，而不是从零开始设计。

### 2.3 长期目标：成为猫墙定制业务的配置中台

在 6-12 个月内，系统升级为业务中台，连接商品库、订单、安装、售后和数据分析。

长期成功标志：

- 每个订单都有完整设计记录、报价记录、施工图和安装反馈。
- 组件销量、模板转化率、常见错误、热门预算段可以被统计。
- 设计系统可以支撑多城市、多安装团队、多价格策略。

## 3. 目标用户与场景

### 3.1 终端客户

客户通常不是来“学习 3D 工具”的，而是想回答几个问题：

- 我家能不能装？
- 装完大概长什么样？
- 会不会挡门、窗、空调、插座？
- 猫能不能安全上下？
- 总价是多少？
- 安装麻不麻烦？

客户模式要尽量减少专业按钮，突出模板、预算、效果图、价格和确认。

### 3.2 设计师/客服

设计师和客服需要的是效率：

- 快速录入墙面尺寸。
- 快速复用模板。
- 根据客户预算替换组件。
- 快速解释为什么某个位置不能放。
- 快速导出报价单、施工图和客户预览图。

设计师模式可以保留高级编辑能力，但需要更好的方案管理和报价能力。

### 3.3 安装师傅

安装师傅不关心 3D 编辑过程，只关心能不能按图施工：

- 每面墙的正视图。
- 每个组件的编号、位置、尺寸、打孔点。
- 从墙边/地面的距离。
- 禁区和避让说明。
- 安装顺序。
- 配件和工具清单。

施工图要从“看起来完整”进一步升级为“现场可执行”。

### 3.4 经营者

经营者关心：

- 哪些模板最容易成交。
- 哪些组件利润高、复购好、安装风险低。
- 哪些错误导致客户放弃。
- 哪些价格区间转化率高。
- 哪些安装问题反复出现。

因此系统最终需要后台和数据能力。

## 6. 商业闭环设计

### 6.1 标准业务流程

```text
客户输入需求
  -> 选择户型/墙面模板
  -> 选择猫数量、体重、预算、风格
  -> 系统推荐方案
  -> 客户或设计师调整
  -> 系统实时校验
  -> 生成报价
  -> 客户确认
  -> 生成施工包
  -> 安装执行
  -> 安装反馈进入数据闭环
```

### 6.2 方案状态

项目需要从“草稿”升级为业务对象，建议状态：

- `draft`：草稿。
- `quoted`：已报价。
- `customer_review`：客户确认中。
- `confirmed`：客户已确认。
- `paid`：已支付。
- `install_scheduled`：已预约安装。
- `installed`：已安装。
- `archived`：归档。

### 6.3 输出物

每个可成交方案至少输出：

- 客户预览图：面向客户，突出效果。
- 报价单：面向成交，包含商品、数量、价格、服务费和有效期。
- 施工图：面向安装，包含位置、尺寸、打孔和安装顺序。
- BOM 清单：面向仓储/发货，包含 SKU、数量、规格和配件。
- JSON 项目文件：面向系统留档和二次编辑。

## 7. 核心功能规划

### 7.1 商品与 SKU Catalog

当前 catalog 应升级为真实商品系统。

每个商品建议包含：

- SKU。
- 商品名称。
- 组件类型。
- 分类和小分类。
- 尺寸。
- 可选颜色。
- 可选材质。
- 承重。
- 适用墙体。
- 是否需要打孔。
- 安装难度。
- 安装耗时。
- 单价。
- 成本价。
- 库存状态。
- 配件包。
- 图片。
- 3D 模型。
- 购买链接。
- 上下架状态。

优先级：

1. SKU、名称、尺寸、价格、模型。
2. 颜色、材质、承重、库存。
3. 配件、安装难度、成本、毛利。

### 7.2 价格与报价

价格不应只是组件价格相加，还应支持完整报价结构：

- 商品小计。
- 标准配件费。
- 安装服务费。
- 远距离服务费。
- 定制加工费。
- 折扣。
- 税费或平台服务费。
- 运费。
- 报价有效期。

第一版可以先做：

- 商品总价。
- 安装费手动输入。
- 运费手动输入。
- 优惠手动输入。
- 总价自动计算。

第二版再做规则化：

- 按城市计算安装费。
- 按组件数量/难度计算人工。
- 按订单金额免运费。
- 按活动策略自动优惠。

### 7.3 安装可行性校验

当前已有禁区和冲突校验，商业化版本要继续升级为“设计规则引擎”。

建议规则：

- 组件不能进入禁区。
- 禁区不能覆盖已有组件。
- 组件不能超出墙面/地面范围。
- 相邻跳板距离不能超过建议值。
- 起点离地高度不能过高。
- 高风险组件需要保留安全缓冲。
- 组件承重必须大于猫体重区间。
- 某些组件不能安装在轻体墙/玻璃墙/瓷砖墙。
- 门窗、空调、插座、踢脚线需要避让。
- 打孔点不能落在禁区内。

规则提示要做到：

- 告诉用户哪里错。
- 告诉用户为什么错。
- 给出可操作建议。

例如：

> 三层跳台距离右侧跳板 82cm，超过建议跳跃距离 60cm。建议向左移动 22cm，或在中间增加一个中跳板。

### 7.4 模板与自动推荐

模板是商业化效率的关键。

建议模板维度：

- 单墙模板。
- 双墙夹角模板。
- 三墙夹角模板。
- 小户型模板。
- 多猫家庭模板。
- 幼猫/老年猫模板。
- 高预算展示模板。
- 租房少打孔模板。
- 窗边观景模板。

自动推荐输入：

- 墙面宽度。
- 墙面高度。
- 猫数量。
- 猫体重。
- 猫年龄。
- 预算。
- 是否可打孔。
- 风格偏好。

输出：

- 低预算方案。
- 标准方案。
- 高配方案。

### 7.5 施工图增强

现有施工图已经是很重要的商业资产。下一步建议补齐：

- 打孔点坐标。
- 组件安装顺序。
- 每个组件的安装方向。
- 配件包编号。
- 安装注意事项。
- 现场复核清单。
- PDF 导出。
- 客户签字确认页。

施工图页面建议分成两种模式：

- 客户版：更干净，展示效果和报价。
- 施工版：更密集，展示坐标、编号、打孔和清单。

### 7.6 项目保存、分享和协作

商业化必须从本地草稿升级到云端项目。

第一阶段：

- 本地项目列表。
- 项目名称、客户姓名、联系方式。
- JSON 导入导出继续保留。

第二阶段：

- 云端保存。
- 分享链接。
- 只读客户链接。
- 设计师编辑链接。
- 客户确认按钮。

第三阶段：

- 多人协作。
- 历史版本。
- 修改记录。
- 评论和批注。

### 7.7 客户模式 UI

客户模式应该隐藏复杂工具，只保留：

- 选择模板。
- 填写墙面尺寸。
- 选择预算。
- 选择猫数量/体重。
- 拖拽替换组件。
- 查看价格。
- 提交方案。

高级能力保留在设计师模式：

- 精确坐标。
- 禁区绘制。
- 施工图。
- 自定义组件。
- 价格细项。
- 项目导入导出。

## 8. 技术路线

### 8.1 前端架构

当前 React + Three.js + Zustand 架构可以继续承载下一阶段。

建议保留分层：

- `domain/scene`：纯业务计算，继续保持可测试。
- `features/scene3d`：3D 编辑体验。
- `features/construction-drawings`：施工图。
- `features/component-palette`：组件选择。
- 新增 `features/quotation`：报价。
- 新增 `features/project-manager`：项目管理。
- 新增 `features/customer-preview`：客户预览。

### 8.2 后端与数据

当前可以先不急着重后端，但商业化必须逐步引入。

阶段一：

- 继续本地存储。
- JSON 可导出。
- 用静态 catalog 验证业务。

阶段二：

- 引入后端 API。
- 项目云端保存。
- 商品库云端管理。
- 分享链接。

阶段三：

- 订单系统。
- 支付或支付跳转。
- 安装排期。
- 数据看板。

### 8.3 数据模型建议

核心对象：

- `Project`：设计项目。
- `Customer`：客户。
- `Room`：空间信息。
- `Plane`：墙面/地面。
- `ComponentInstance`：已放置组件。
- `ForbiddenZone`：禁区。
- `ProductSku`：商品 SKU。
- `Quote`：报价。
- `QuoteItem`：报价行。
- `ConstructionPackage`：施工包。
- `Order`：订单。

## 10. MVP 范围建议

第一版商业化 MVP 应该克制，不要一次做完整平台。

### 必做

- 真实 SKU catalog。
- 实时商品总价。
- 报价单导出。
- 施工图 PDF。
- 客户/项目基础信息。
- 方案保存和复制。
- 规则提示优化。
- 10 套高质量模板。

### 可以后置

- 在线支付。
- 多人协作。
- 复杂权限。
- AI 自动布局。
- 大型数据看板。
- 多城市价格体系。
- 完整库存系统。

### 暂不建议做

- 通用家装设计平台。
- 过度复杂的自由建模。
- 面向所有宠物品类。
- 社区/内容平台。
- 复杂 CAD 级绘图能力。

## 11. 商业模式

### 11.1 自营猫墙定制

编辑器作为内部销售和交付工具，提高转化和效率。

收入：

- 猫墙组件销售。
- 定制设计服务费。
- 安装服务费。
- 配件和耗材。

优势：

- 最容易验证。
- 业务闭环短。
- 数据积累快。

### 11.2 SaaS 给猫墙/宠物家居商家

将编辑器提供给其他商家，用于他们自己的商品和报价。

收入：

- 月费。
- 按项目收费。
- 按订单抽成。
- 高级模板/高级导出收费。

前提：

- 商品库、价格规则、品牌化配置要成熟。
- 需要账号、权限、云端和后台。

### 11.3 组件供应链平台

长期可以连接多个猫墙组件品牌，让用户设计后直接采购。

收入：

- 商品佣金。
- 流量分发。
- 设计服务。

前提：

- 要有足够用户流量。
- 商品标准化程度要高。

建议优先顺序：

1. 自营成交工具。
2. 半自助客户配置器。
3. SaaS。
4. 供应链平台。

## 12. 关键指标

### 12.1 设计效率指标

- 平均出初版方案时间。
- 平均修改次数。
- 平均报价时间。
- 每位设计师每日可处理方案数。

### 12.2 转化指标

- 咨询到报价转化率。
- 报价到成交转化率。
- 平均客单价。

### 12.3 交付指标

- 施工图返工率。
- 安装现场问题率。
- 漏发/错发组件率。
- 客户确认后修改率。
- 安装完成满意度。

### 12.4 商品指标

- 组件使用频率。
- 组件成交频率。
- 模板转化率。
- 组件毛利。
- 高风险组件问题率。

## 13. 风险与应对

### 13.1 设计自由度过高导致客户不会用

应对：

- 客户模式模板优先。
- 默认推荐三套方案。
- 高级编辑隐藏到设计师模式。

### 13.2 3D 看起来对，施工现场仍然装不了

应对：

- 强化墙体类型、禁区、打孔点校验。
- 增加现场复核清单。
- 安装反馈回流规则库。

### 13.3 商品数据不标准，报价不可信

应对：

- 先整理 SKU 标准。
- 报价规则第一版允许人工调整。
- 报价单保留“待确认项”。

### 13.4 功能越做越像通用 CAD，失去垂直优势

应对：

- 坚持猫墙场景。
- 所有功能围绕成交、报价、施工。
- 不追求通用建模能力。

### 13.5 真实 3D 模型制作成本高

应对：

- 先保证核心畅销 SKU 有真实模型。
- 低频 SKU 使用标准占位模型。
- 模型资产制作纳入商品上架流程。

## 14. 团队配置建议

### MVP 阶段

- 产品/业务负责人 1 人：定义报价、SKU、流程和验收。
- 前端/全栈工程 1-2 人：编辑器、报价、施工图、项目管理。
- 设计/视觉 0.5-1 人：客户模式、报价单、施工图视觉。
- 商品/运营 1 人：整理 SKU、模板、价格。
- 安装顾问 1 人兼职：校验施工图和规则。

### 平台阶段

- 前端工程。
- 后端工程。
- 3D/图形工程。
- 产品经理。
- UI/UX 设计。
- 运营后台负责人。
- 数据分析。

## 15. 近期可执行任务清单

### 第 1 周

- 梳理 30-50 个真实 SKU。
- 给每个 SKU 补齐价格、尺寸、分类、安装属性。
- 选出 10 个高转化模板。
- 定义报价单字段。
- 定义施工图 PDF 结构。

### 第 2-3 周

- 开发报价面板。
- 开发报价单导出。
- 完善施工图 PDF。
- 给项目增加客户信息。
- 增加方案复制。

### 第 4 周

- 用 10 个真实场景内测。
- 记录每个方案耗时、错误、返工点。
- 优化规则提示。
- 优化模板。

### 第 5-6 周

- 上线客户预览页。
- 支持分享链接。
- 支持客户确认/留言。
- 将内部成交流程跑通。

## 16. 建议的下一步

建议下一步不要直接做“大平台”，而是先做“内部成交工具 MVP”。

最小闭环是：

```text
真实 SKU
  -> 设计方案
  -> 自动报价
  -> 施工图 PDF
  -> 客户确认
  -> 安装执行
```

只要这个闭环跑通，编辑器就从“好用的工具”变成“能产生收入的系统”。

## 17. 总结

当前编辑器最有商业价值的地方，是它已经连接了三个关键环节：

- 3D 方案设计。
- 规则化摆放校验。
- 2D 施工图与 BOM。

后续商业化的主线应该围绕“成交和交付”展开，而不是无限增加编辑功能。优先把商品、价格、报价、施工、客户确认这些业务对象做扎实，再逐步扩展客户自助、后台运营和智能推荐。

商业化版本的第一目标不是做一个最强 3D 编辑器，而是做一个能让猫墙业务更快成交、更少返工、更容易交付的系统。
