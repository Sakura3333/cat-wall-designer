## 1. Problem Statement

现有 AI CAD 校准页以表单为主。用户需要把商品图中的整体尺寸、板件尺寸、深度、孔径和孔位逐项翻译为字段，再依靠脑内想象判断这些数值是否组成了正确结构。这个方式虽然容易实现和验证，但实际使用不直观，尤其不适合包含多个板件、孔和规则曲面的组件。

新的目标是让 Vision AI 先根据商品图生成一个可编辑的参数化三视图草稿。用户直接在正视图、俯视图和右视图中选择、增删和修改受支持形状，点击尺寸标注输入精确毫米值，并通过实时 3D 预览检查结构。草图确认后再生成下游 `ComponentSpec`，进入 Planner、Blender Worker 和 QA 流程。

系统的核心不是三张独立的二维图，而是一组带尺寸、位置、变换和约束的三维参数实体。三视图只是同一数据源的联动投影。

## 2. Demand Evidence

- 当前组件由一名内部使用者在 Blender 中手工建模，平均约 10 分钟一个，已有约 30 个待建组件，组件库还会持续增长。
- 用户已经实际使用了首版校准页，并明确反馈：“表单填写参数的形式整体上使用下来，不太直观”。
- 用户提出的替代工作流非常具体：从商品图生成三视图草稿，在草稿中增删改形状、修改线条长度、直接填写形状尺寸和位置，再进入后续建模流程。
- 组件只用于 Web 3D 展示，但整体尺寸、板件尺寸和施工孔位需要毫米级一致；这要求编辑体验直观，同时事实数据仍必须结构化和可验证。

## 3. Status Quo

当前流程如下：

```text
商品图证据
  -> 表单填写整体尺寸
  -> 表格填写每块板件 X/Y/Z
  -> 表单填写孔径、孔中心和轴向
  -> 阻断项全部解除
  -> ComponentSpec v1
```

主要问题：

1. 用户需要在商品图、字段名称和脑内三维结构之间来回转换。
2. 数值正确不代表部件位置关系正确，当前界面无法直接发现装配错误。
3. 表单把尺寸和位置拆散，无法表达“共面、镜像、同心、对齐”等结构关系。
4. 当前 `ComponentSpec v1` 只有部件尺寸，没有部件位置和旋转，不能完整表达多部件装配。
5. AI 给出的建议值和用户最终确认的几何之间缺少一个直观、可编辑、可追踪的中间产物。

## 4. Target User And Narrowest Wedge

### 4.1 首版用户

公司内部唯一的组件建模人员。该用户熟悉 Blender 和组件结构，但不应被要求手写 JSON、理解坐标转换或在多个表单之间推导装配关系。

### 4.2 最窄可用版本

首版只需要把“太空舱组件”从商品图草稿校准到可生成 `ComponentSpec`，支持：

- 矩形板件；
- 盒体；
- 圆柱；
- 半球；
- 一种明确参数化的圆柱扇区曲面板；
- 施工孔；
- 轴对齐的位置和尺寸；
- 90 度整数倍旋转；
- 对齐、共面、镜像、固定距离和包围盒约束。

不做通用 CAD，不支持任意贝塞尔曲线、不规则曲面、自由曲面、草图拉伸任意轮廓、装配运动学和制造级公差链。首个生产闭环只针对太空舱的 5 个实体和有限施工孔，通用多产品族能力在闭环证明后扩展。

## 5. Agreed Premises

1. **三视图必须联动。** 修改任一视图中的实体尺寸或位置，其他视图和 3D 预览同步更新。
2. **三维实体是事实源。** 正视图、俯视图和右视图不能独立保存互相矛盾的几何。
3. **首版采用受约束编辑。** 用户只能创建和编辑后续编译器明确支持的参数化形状。
4. **AI 负责起草，不负责批准。** 缺失尺寸可以用于视觉近似，但不能自动成为已确认硬约束。
5. **毫米是业务单位。** 草图、规格、计划和 QA 全部使用毫米；Three.js 显示层自行换算单位。
6. **表单降级为属性检查器。** 精确数值仍可在右侧属性面板编辑，但不再是主要工作流。
7. **本地 Worker 的部署拓扑不变。** 草图编辑器不会改变 Worker 必须本地安装、主动领取任务的方式，但会升级 `ComponentSpec`、`ModelingPlan` 和 Blender Compiler 的数据契约与节点能力。

## 6. Approaches Considered

### 6.1 Approach A: 在现有表单上叠加 SVG 三视图

- 工作量：中，约 5-7 天。
- 风险：低。
- 优点：复用当前太空舱草稿结构，上线最快。
- 缺点：数据结构与单个样本绑定；位置、变换和约束会散落在 UI 状态中；后续产品族需要重构。

### 6.2 Approach B: 独立参数化草图引擎（已批准）

- 工作量：大；太空舱确定性草图到 Worker 闭环约 5-6 周，加入 Vision 后约 6-8 周，通用化约 8-10 周。
- 风险：中。
- 优点：所有产品族共用一个中间规格；投影、约束和编译可以独立测试；可逐步扩充形状而不重写页面。
- 缺点：必须先定义新的数据合同和命令模型，首个可见版本比 Approach A 晚。

### 6.3 Approach C: Three.js 三维对象优先

- 工作量：大。
- 风险：中高。
- 优点：三视图一致性天然成立，直接操作三维对象。
- 缺点：工程图式尺寸线、二维捕捉、孔位标注、打印和鼠标命中更难；不适合作为二维精确编辑的主要实现。

## 7. Recommended Architecture

采用 Approach B，但按“太空舱生产闭环 -> 通用化”拆分。首个版本不追求一次完成通用 CAD 引擎。整体数据流：

```text
商品图
  -> VisionObservation v1
  -> AI Draft Adapter
  -> OrthographicSketchSpec v1
  -> 三视图校准 + 3D 预览
  -> Sketch Compiler
  -> ComponentSpec v2
  -> Planner
  -> ModelingPlan v2
  -> Local Worker + Blender 4.5 LTS
  -> GLB + Blend + QA
```

职责边界：

- `VisionObservation` 保存图片证据、识别候选、置信度和歧义。
- `OrthographicSketchSpec` 保存用户正在校准的三维参数实体、证据绑定、确认状态和约束。
- 三视图渲染器只负责投影、命中测试、控制柄和尺寸标注。
- 约束引擎负责确定性传播和冲突检测。
- `Sketch Compiler` 只接受无阻断项的草图，生成版本化 `ComponentSpec`。
- Planner 不再从图片猜测已确认数据，只把规格编译为建模操作。

## 8. Coordinate System

统一采用右手坐标系：

- `+X`：组件向右，表示宽度；
- `+Y`：组件向上，表示高度；
- `+Z`：组件朝前，表示深度；
- 原点：持久化为组件局部包围盒中心；最终导出使用同一原点，不在编译时静默重心化；
- 单位：毫米。

视图映射：

| 视图 | 水平轴 | 垂直轴 | 被压缩轴 |
| --- | --- | --- | --- |
| 正视图 | X | Y | Z |
| 右视图 | Z | Y | X |
| 俯视图 | X | Z | Y |

用户在任一视图拖动控制柄时，编辑器先把屏幕位移逆投影到对应业务轴，再更新同一个三维实体。其他视图不保存重复坐标，只重新投影。任何导出原点变换都必须作为显式 `exportTransform` 记录。

## 9. OrthographicSketchSpec V1

### 9.1 顶层结构

```ts
type OrthographicSketchSpecV1 = {
  schemaVersion: 1
  revision: number
  revisionId: string
  unit: 'mm'
  jobId: string
  name: string
  family: string
  placement: 'wall' | 'floor' | 'free'
  coordinateSystem: {
    handedness: 'right'
    up: '+Y'
    front: '+Z'
    originPolicy: 'component-local-center'
  }
  overallEnvelope: SketchEnvelope
  entities: SketchEntityV1[]
  holes: SketchHoleV1[]
  constraints: SketchConstraintV1[]
  annotations: SketchAnnotationV1[]
  assumptions: SketchAssumptionV1[]
  viewState?: OrthographicViewState
}

type Vec3 = { x: number; y: number; z: number }
type Axis = '+X' | '-X' | '+Y' | '-Y' | '+Z' | '-Z'

type SketchEnvelope = {
  sizeMm: Vec3
  parameterState: { x: ParameterState; y: ParameterState; z: ParameterState }
}

type OrthographicViewState = {
  zoom: number
  panMm: { x: number; y: number }
  gridMm: 0.5 | 1 | 5 | 10
  visibleEntityIds: string[]
}

type ParameterStateBase = {
  source: 'vision' | 'image-dimension' | 'engineering-drawing' | 'user' | 'template'
  evidenceRefs: string[]
}

type ParameterState =
  | (ParameterStateBase & { status: 'proposed' })
  | (ParameterStateBase & { status: 'confirmed' | 'approved-default'; confirmedBy: string; confirmedAt: string })

type SketchAnnotationV1 = {
  id: string
  kind: 'dimension' | 'centerline' | 'callout'
  entityId?: string
  parameter?: string
  valueMm?: number
  view: 'front' | 'right' | 'top'
  anchor: { u: number; v: number }
  status: 'proposed' | 'confirmed' | 'rejected'
}

type SketchAssumptionV1 = {
  field: string
  reason: string
  severity: 'info' | 'warning' | 'blocking'
  status: 'open' | 'accepted' | 'rejected'
}
```

`viewState` 只保存缩放、平移、网格和显示开关，不参与建模事实，也不进入 Planner。

### 9.2 通用实体字段

```ts
type SketchEntityBase = {
  id: string
  label: string
  primitive: SketchPrimitiveKind
  transform: {
    positionMm: { x: number; y: number; z: number }
    rotationDeg: { x: number; y: number; z: number }
  }
  materialRole: string
  hardConstraint: boolean
  parameterState: Record<string, ParameterState>
  notes?: string
}
```

### 9.3 首版实体联合类型

```ts
type SketchEntityV1 =
  | (SketchEntityBase & {
      primitive: 'rect-panel' | 'box'
      sizeMm: { x: number; y: number; z: number }
      thicknessAxis?: 'x' | 'y' | 'z'
    })
  | (SketchEntityBase & {
      primitive: 'round-panel' | 'cylinder'
      diameterMm: number
      depthMm: number
      axis: '+X' | '-X' | '+Y' | '-Y' | '+Z' | '-Z'
    })
  | (SketchEntityBase & {
      primitive: 'hemisphere'
      diameterMm: number
      shellThicknessMm: number
      facing: '+X' | '-X' | '+Y' | '-Y' | '+Z' | '-Z'
    })
  | (SketchEntityBase & {
      primitive: 'curved-panel'
      radiusMm: number
      sweepDeg: number
      axialLengthMm: number
      thicknessMm: number
      axis: '+X' | '+Y' | '+Z'
      startAngleDeg: number
    })
  | {
      id: string
      kind: 'unsupported'
      label: string
      sourcePolygon?: Array<[number, number]>
      reason: string
      evidenceRefs: string[]
      status: 'blocking'
    }
```

任意路径、自由多边形和贝塞尔控制点不进入 V1。

所有尺寸参数都定义在实体局部坐标中，并在 `rotationDeg` 之前计算：Box 和板件以局部中心为原点；圆柱以轴向中点为原点；半球以球心为原点，`facing` 指向球冠极点并占据该轴正半空间；圆柱扇区曲面以母圆圆心和轴向中点为原点，`startAngleDeg` 和 `sweepDeg` 在朝向轴正方向观察时逆时针增加，材料占据 `[radiusMm - thicknessMm, radiusMm]`。`axis`/`facing` 是实体局部轴，随后才应用仅允许 90 度整数倍的实体旋转。

V1 半球固定为薄壳，`shellThicknessMm` 是必填且必须确认的硬参数；不允许省略后按实心几何解释。

`parameterState` 的 key 使用稳定字段路径，例如 `sizeMm.x`、`transform.positionMm.z`、`diameterMm`。Schema 为每种 primitive 枚举必需 key，不能接受任意未声明字段。

### 9.4 施工孔

```ts
type SketchHoleBaseV1 = {
  id: string
  label: string
  targetEntityId: string
  entryPointLocalMm: Vec3
  entryFace: { axis: 'x' | 'y' | 'z'; side: 'min' | 'max' }
  coordinateSpace: 'entity-local'
  axisLocal: Axis
  diameterMm: number
}

type HoleCommonParameterState = Record<'entryPointLocalMm.x' | 'entryPointLocalMm.y' | 'entryPointLocalMm.z' | 'entryFace' | 'axisLocal' | 'diameterMm' | 'through', ParameterState>

type SketchHoleV1 = SketchHoleBaseV1 & (
  | { through: true; depthMm?: never; parameterState: HoleCommonParameterState }
  | { through: false; depthMm: number; parameterState: HoleCommonParameterState & { depthMm: ParameterState } }
)
```

孔必须锚定目标实体。`entryPointLocalMm` 是钻孔轴与 `entryFace` 的交点，`axisLocal` 必须从入口面指向实体内部。移动目标实体时，入口点的局部位置保持不变；用户可以切换到组件坐标检查最终绝对位置。`through === true` 时禁止提供 `depthMm`；盲孔必须为 `through === false` 且提供正的 `depthMm`。编译和 QA 时再通过实体变换转换到组件坐标。V1 孔只允许放在轴对齐平面板上，曲面孔位后置。

### 9.5 约束

首个太空舱生产闭环只支持以下单向、无环约束：

- `align-axis`：实体中心或边沿在 X/Y/Z 轴对齐；
- `coincident-plane`：两个板面共面；
- `mirror`：指定 master 实体围绕 X/Y/Z 平面对称创建或更新 slave 实体；
- `fixed-distance`：两个点、中心或平面间距离固定；
- `inside-envelope`：实体必须位于整体包围范围内；
- `hole-on-target`：孔轴必须穿过目标实体的有效范围。

约束合同固定为：

```ts
type ConstraintParameterPath =
  | 'transform.positionMm.x' | 'transform.positionMm.y' | 'transform.positionMm.z'
  | 'sizeMm.x' | 'sizeMm.y' | 'sizeMm.z'
  | 'diameterMm' | 'depthMm' | 'radiusMm' | 'axialLengthMm' | 'thicknessMm'

type ConstraintRef =
  | { kind: 'parameter'; entityId: string; path: ConstraintParameterPath }
  | { kind: 'face'; entityId: string; axis: 'x' | 'y' | 'z'; side: 'min' | 'center' | 'max' }
  | { kind: 'hole-center'; holeId: string; axis: 'x' | 'y' | 'z' }

type SketchConstraintV1 =
  | { id: string; kind: 'align-axis'; driver: ConstraintRef; driven: ConstraintRef[]; axis: 'x' | 'y' | 'z'; offsetMm: number; toleranceMm: number; status: ConstraintStatus }
  | { id: string; kind: 'coincident-plane'; driver: ConstraintRef; driven: ConstraintRef[]; toleranceMm: number; status: ConstraintStatus }
  | { id: string; kind: 'mirror'; masterEntityId: string; slaveEntityId: string; plane: { axis: 'x' | 'y' | 'z'; offsetMm: number }; toleranceMm: number; status: ConstraintStatus }
  | { id: string; kind: 'fixed-distance'; driver: ConstraintRef; driven: ConstraintRef; axis: 'x' | 'y' | 'z'; distanceMm: number; toleranceMm: number; status: ConstraintStatus }
  | { id: string; kind: 'inside-envelope'; entityId: string; marginMm: number; toleranceMm: number; status: ConstraintStatus }
  | { id: string; kind: 'hole-on-target'; holeId: string; targetEntityId: string; toleranceMm: number; status: ConstraintStatus }

type ConstraintStatus = 'active' | 'conflict' | 'suppressed'
```

规则按 `driver -> driven` 的拓扑顺序执行，V1 禁止约束环和两个 driver 写同一参数。拖动 driven 参数时拒绝编辑并提示 driver；拖动 driver 时原子更新所有 driven。检测到矛盾时不自动选赢家，而是进入冲突状态。`equal-parameter`、`concentric` 和通用非线性求解器后置。

`face` 和 `hole-center` 引用在应用实体 90 度旋转后转换到组件坐标；`parameter` 引用仍指实体局部标量。合法组合固定为：`align-axis` 接受同一组件轴上的 parameter/face/hole-center；`coincident-plane` 只接受两个 face；`fixed-distance` 接受同一组件轴上的两个引用；其他约束使用专用 ID 字段。非法组合在 Schema 后的语义校验阶段拒绝。

太空舱版本的 mirror 只允许 `rect-panel` 和 `box`。反射会变换组件坐标位置和 90 度旋转，复制 master 的局部形状参数；附属孔根据镜像平面变换入口点、入口面和轴向并生成新 ID。半球、圆柱和曲面板的 mirror 在定义完整朝向映射前保持阻断，不允许近似处理。

## 10. ComponentSpec V2 Requirement

当前 `ComponentSpec v1` 的部件只有 `shape` 和 `sizeMm`，缺少装配所必需的 `positionMm` 和 `rotationDeg`。三视图草图不能可靠编译到该结构；现有 `ModelingPlan v1` 也只支持 Box 和 mounting-hole，无法承载半球、圆柱和圆柱扇区曲面。

因此需要新增 `ComponentSpec v2`：

```ts
type ComponentPartBaseV2 = {
  id: string
  label: string
  transform: {
    positionMm: Vec3
    rotationDeg: Vec3
  }
  hardConstraint: boolean
  materialRole: string
  parameterState: Record<string, ParameterState>
}

type ComponentPartV2 = ComponentPartBaseV2 & (
  | { shape: 'rect-panel' | 'box'; parameters: { sizeMm: Vec3; thicknessAxis?: 'x' | 'y' | 'z' } }
  | { shape: 'round-panel' | 'cylinder'; parameters: { diameterMm: number; depthMm: number; axis: Axis } }
  | { shape: 'hemisphere'; parameters: { diameterMm: number; shellThicknessMm: number; facing: Axis } }
  | { shape: 'curved-panel'; parameters: { radiusMm: number; sweepDeg: number; axialLengthMm: number; thicknessMm: number; axis: '+X' | '+Y' | '+Z'; startAngleDeg: number } }
)

type ComponentHoleBaseV2 = {
  id: string
  partId: string
  entryPointLocalMm: Vec3
  entryFace: { axis: 'x' | 'y' | 'z'; side: 'min' | 'max' }
  axisLocal: Axis
  diameterMm: number
}

type ComponentHoleV2 = ComponentHoleBaseV2 & (
  | { through: true; depthMm?: never; parameterState: HoleCommonParameterState }
  | { through: false; depthMm: number; parameterState: HoleCommonParameterState & { depthMm: ParameterState } }
)

type ComponentSpecV2 = {
  schemaVersion: 2
  revision: number
  unit: 'mm'
  name: string
  family: string
  placement: 'wall' | 'floor' | 'free'
  overallSizeMm: Vec3
  parts: ComponentPartV2[]
  holes: ComponentHoleV2[]
  assumptions: SketchAssumptionV1[]
  source: { jobId: string; sketchRevision: number; sketchRevisionId: string; sourceImageSha256: string }
}
```

部件和孔的整体 readiness 由必需 `parameterState` 计算，不持久化第二份聚合状态。`confirmed` 和 `approved-default` 的硬参数必须同时具有非空 `confirmedBy` 与 `confirmedAt`；否则语义校验失败。未经人工批准的模板值只能保持 `proposed`。

`ModelingPlan v2` 完整合同：

```ts
type PlanTransform = { positionMm: Vec3; rotationDeg: Vec3 }

type ModelingPlanNodeV2 =
  | { id: string; op: 'box'; partId: string; sizeMm: Vec3; transform: PlanTransform; materialRole: string }
  | { id: string; op: 'cylinder'; partId: string; diameterMm: number; depthMm: number; axisLocal: Axis; transform: PlanTransform; materialRole: string }
  | { id: string; op: 'hemisphere'; partId: string; diameterMm: number; shellThicknessMm: number; facingLocal: Axis; transform: PlanTransform; materialRole: string }
  | { id: string; op: 'curved-panel'; partId: string; radiusMm: number; sweepDeg: number; axialLengthMm: number; thicknessMm: number; axisLocal: '+X' | '+Y' | '+Z'; startAngleDeg: number; transform: PlanTransform; materialRole: string }
  | ({ id: string; op: 'mounting-hole'; holeId: string; targetNodeId: string; entryPointLocalMm: Vec3; entryFace: { axis: 'x' | 'y' | 'z'; side: 'min' | 'max' }; axisLocal: Axis; diameterMm: number } & ({ through: true; depthMm?: never } | { through: false; depthMm: number }))

type ModelingPlanV2 = {
  schemaVersion: 2
  componentSpecRevision: number
  componentSpecRevisionId: string
  coordinateSystem: { handedness: 'right'; up: '+Y'; front: '+Z'; unit: 'mm'; originPolicy: 'component-local-center' }
  nodes: ModelingPlanNodeV2[]
  export: { primary: 'glb'; optional: Array<'blend' | 'obj' | 'fbx' | 'stl'>; originPolicy: 'component-local-center'; applyTransforms: true }
}
```

对应 JSON Schema 使用 `oneOf` 表达节点和 through/blind hole 判别联合，所有对象 `additionalProperties: false`，所有 ID 使用 stable ID pattern，且把 `targetNodeId` 存在性、目标先于孔节点和无悬空引用作为语义校验。

确定性映射规则：每个 `ComponentPartV2` 生成一个同 ID 派生的几何节点；节点按 part stable ID 排序。每个孔在其 target node 之后生成 mounting-hole 节点，并按 hole stable ID 排序。Planner 不得合并、丢弃或重解释已确认参数；它只验证节点能力并生成上述映射。执行顺序就是拓扑排序后的数组顺序，孔的 `targetNodeId` 必须引用先前的几何节点。所有 transform 先作用于实体几何，孔 cutter 在目标实体局部坐标构造后随目标 transform 进入组件坐标。`ModelingPlan v1` 继续只服务 Phase 0 Fixture。

迁移策略：

1. 保留 `ComponentSpec v1` Schema 和 Phase 0 Fixture，确保已有 Worker 回归不受影响。
2. 新建 `ComponentSpec v2` Schema、类型和语义校验。
3. 为只包含单个原点 Box 的 v1 提供兼容升级器，默认位置和旋转为零。
4. Planner 和 Blender Compiler 逐步支持 v2；不允许把多部件 v2 降级为丢失位置的 v1。

### 11.2 主要操作

- 单击实体：三个视图和 3D 预览中同步高亮同一实体。
- 拖动实体：修改当前视图对应的两个坐标轴。
- 拖动控制柄：修改当前视图可见的尺寸参数。
- 单击尺寸数字：原位进入毫米输入；提交后立即重新投影。
- 添加形状：从受支持形状菜单创建实体；不可见轴参数标为待确认。

- 添加孔：选择目标实体和可见板面后放置孔中心，再输入孔径、深度和轴向。
- 镜像：创建显式 master/slave `mirror` 约束；master 可编辑，slave 的受约束参数只读。
- 3D 预览：仅用于检查结构、遮挡和朝向，不作为毫米尺寸的唯一输入面。

## 12. Editing State Machine

```text
loading-ai-draft
  -> draft-ready
  -> editing
  -> validating
  -> blocked | ready-for-approval
  -> approved
  -> compiling-spec
  -> spec-ready | compile-failed
```

冲突是 `editing` 的子状态：

```text
editing
  -> constraint-conflict
  -> suppress-remove-or-reverse-constraint
  -> editing
```

V1 不允许临时选择多个 driver。冲突只能通过抑制约束、删除约束或执行 `ReverseConstraintDriverCommand` 反转一个合法的单向约束来解决。

所有命令写入可撤销命令日志：

- `AddEntityCommand`；
- `DeleteEntityCommand`；
- `MoveEntityCommand`；
- `SetParameterCommand`；
- `SetRotationCommand`；
- `AddHoleCommand`；
- `SetHoleParameterCommand`；
- `AddConstraintCommand`；
- `RemoveConstraintCommand`；
- `SuppressConstraintCommand`；
- `ReverseConstraintDriverCommand`；
- `ConfirmParameterCommand`。

每个命令保存执行前后的最小完整快照；suppress 的逆操作恢复原状态，reverse 的逆操作恢复原 driver/driven 和传播结果。连续拖动只在指针释放时合并为一个历史命令，避免撤销栈被每一帧填满。

## 15. Validation And Blocking Rules

草图只有满足以下条件才可生成 `ComponentSpec v2`：

1. 整体 X/Y/Z 包围尺寸均为正值并已确认或明确批准默认值。
2. 每个实体的必需形状参数完整且为正值。
3. 每个实体的位置和允许的旋转完整。
4. 硬约束参数不得处于 `proposed` 状态。
5. 所有施工孔的目标实体、局部中心、局部轴向和孔径完整且已确认。
6. 孔入口点位于目标面有效范围，入口面、轴向和深度语义一致。
7. 不存在悬空约束、循环主导约束或未解决冲突。
8. 每个视图投影包围盒与三维实体计算结果一致。
9. V1 中实体世界包围盒并集必须与已确认 `overallEnvelope` 在 1 mm 内一致，不允许例外；组件原点就是该 envelope 的中心。
10. 所有 stable ID 唯一并满足 Schema 规则。

平面孔入口点到目标面每条边的距离必须至少为 `diameterMm / 2 + 0.5 mm`；与边相切或越界均为阻断错误。盲孔深度不得超过沿 `axisLocal` 到目标实体背面的距离。

视觉近似尺寸可以参与 3D 预览，但必须显示为虚线或不同状态，不得解除硬阻断项。所有数值必须为有限数；尺寸、半径、直径、厚度、深度和 envelope 必须为正数，位置、90 度旋转、偏移和起始角允许为零或负数。曲面板 sweep 必须在 `(0, 360]` 且 `0 < thicknessMm < radiusMm`；半球必须满足 `0 < shellThicknessMm < diameterMm / 2`；旋转必须是 90 度整数倍。镜像或复制生成新 stable ID；删除实体的级联结果必须包含在同一个可撤销命令中。`ComponentPartV2` 的判别联合保证 shape 与 parameters 兼容。

QA 的 1 mm 误差使用 Blender evaluated dependency graph 的世界包围盒和实际布尔结果测量，不使用渲染像素估计。板件、圆柱和半球使用解析参数与 evaluated mesh 包围盒交叉检查；曲面网格检查解析参数和世界包围盒。孔 QA 沿预期轴对 evaluated mesh 做入口和出口截面，测量实际开口中心、直径、轴向和盲孔终点；解析孔参数只作为 expected value，不能代替实际网格测量。Compiler 内部目标容差为 0.5 mm，发布硬阻断为 1 mm。

## 16. AI Draft Contract

Vision/LLM 不直接生成 SVG，也不生成任意路径。模型输出 `OrthographicSketchDraft` 结构化数据：

- 产品族候选；
- 实体候选及形状类型；
- 参数建议值；
- 局部位置和轴向；
- 参数与图片证据的关联；
- 每个参数的置信度；
- 未识别或不可见参数列表；
- 建议约束。

最小 Draft 合同：

```ts
type DraftParameter<T> = {
  value?: T
  confidence: number
  evidenceRefs: string[]
  missingReason?: 'occluded' | 'not-labeled' | 'ambiguous' | 'unsupported'
}

type OrthographicSketchDraftV1 = {
  schemaVersion: 1
  jobId: string
  sourceObservationRevision: number
  generator: { provider: string; model: string; promptVersion: string }
  envelope: { x: DraftParameter<number>; y: DraftParameter<number>; z: DraftParameter<number> }
  items: Array<
    | { kind: 'supported'; id: string; label: string; primitive: 'rect-panel' | 'box' | 'round-panel' | 'cylinder' | 'hemisphere' | 'curved-panel'; parameters: Record<string, DraftParameter<unknown>>; transform: Record<string, DraftParameter<number>> }
    | { kind: 'unsupported'; id: string; label: string; reason: string; evidenceRefs: string[] }
  >
  holes: Array<{ id: string; targetCandidateId?: string; parameters: Record<string, DraftParameter<unknown>> }>
  ambiguities: Array<{ code: string; message: string; severity: 'warning' | 'blocking' }>
}
```

Adapter 对 supported item 只生成 `proposed` 参数；缺失参数生成 blocking assumption；unsupported item 原样进入阻断占位符。模型调用只发生在服务端 Provider Adapter，浏览器不持有模型密钥。原始响应、模型、promptVersion 和 Schema 版本必须随任务保存。Vision 接入前建立至少 12 张黄金图片评测集，分别统计实体召回率、参数直接接受率、平均人工修正字段数和校准耗时。

服务端使用 JSON Schema 验证后再进入 Web 编辑器。任何未知形状进入显式 `unsupported` 草稿项，只显示证据和阻断项，不允许模型注入 SVG、脚本或 Blender Python。单张斜视图不能可靠推断被遮挡深度；没有尺寸证据的深度只能作为 proposed，必须由用户确认。

## 17. Persistence And Revision

- 每次用户命令更新内存草稿，并进行节流后的本地持久化；本地存储只是可丢失的恢复缓存。
- 正式 API 接入后，关键审批、生成 Spec 和提交 Worker 时创建不可变 revision。
- 草稿保存当前 `revision`、父 revision 和命令摘要。
- 正式 API 中已批准的 revision 不原地修改；后续编辑创建新 revision。内部首版在 API 接入前只保存草稿快照和审批时间，不把 localStorage 当作审计存储。
- `ComponentSpec v2` 保存来源 `sketchRevision`，QA 可追溯到具体草图。
- 首版内部模式可继续使用浏览器本地存储；正式任务系统接入后迁移到 API 和数据库。

## 18. Error Handling

- AI 草稿失败：创建空的受约束草图并保留商品图证据。
- 部分实体不支持：支持实体正常编辑，不支持实体显示阻断占位符。
- 约束冲突：高亮冲突实体和尺寸，不自动选择赢家。
- 编译失败：保留草图 revision，显示精确到实体和字段的错误。
- 3D 预览失败：不丢草图，允许继续二维校准，但禁止最终视觉审批。
- 本地 Worker 离线：草图和 `ComponentSpec` 仍可保存，任务进入 `waiting_for_device`。
- 刷新或崩溃恢复：从最后一次本地草稿恢复，并提示是否恢复未提交编辑。

## 19. Performance Targets

- 首个闭环最多 30 个实体和孔；通用化后根据真实样本重新设定上限；
- 单次参数修改到三个 SVG 视图更新低于 50 ms；
- 拖动过程保持可感知流畅，目标 60 FPS，最低不低于 30 FPS；
- 3D 预览更新可合并到下一动画帧，不阻塞尺寸输入；
- 命令历史默认保留最近 200 步；
- 自动保存不得阻塞主线程。

## 26. The Assignment

在编码 Milestone 1 前，为“太空舱直径 30”准备一份完整真值：

- 整体 X/Y/Z；
- 五个实体的形状参数；
- 每个实体的中心位置和旋转；
- 所有施工孔的目标板件、孔径、中心和轴向；
